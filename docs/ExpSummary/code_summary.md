> 一些业务开发的经验总结(没啥用)

- 这种业务逻辑，需要尽可能的融入原本数据流；
- 脚本调用，考虑异常情况，必要情况下，添加try catch保护；
- 整理一些C++ 严谨的写法，const修饰orrvide等；
- 结合AI辅助去理解&整理需求，比实现业务代码更加重要；
  - 直接让ai去整理这个图，我觉得意义不大，而结合ai去理解需求，整理流程比较关键，至于真正的代码实现，对于这种简单的脚本或者业务代码，就不是很重要了；

> 交付软件，缺少沉淀；
- 对于“大刀阔斧”的改动，似乎不太容易被接受；
- 对于从代码设计层面就有问题的逻辑，我并不想再已经有问题的结构上缝缝补补；

---

> 工程设计角度思考改动点

- 有时候需要以工程设计角度考虑改动，即便是对于小脚本的修改；
  - 例如一个对于ota升级的脚本，可以升级SOC、MCU、Kernel等，在添加升级Kernel逻辑时；
    - 对于版本校验，与升级无关的判断放到函数外，例如当前SOC芯片版本；
    - 与Kernel版本有关的校验放到函数内部，例如目标版本与当前版本；

一个很小的思考，甚至如果你不把它单拎出来，都感觉不到思考&设计。

进一步去想，在AI时代，这个反而才是更加需要人力去做的事情，即设计实现方式。

---

> 业务项目长久维护

- 在当前架构内，尽可能地考虑扩展性和统一风格，即便是对于一些小的改动；
  - 这样才能在不太容易接受大改动的现有框架内，尽可能的有条理地、有规范地长久维护；

> AI Agent工具的思考

- 还是要自己掌握主动权，不能慢慢的全部交由AI；
  - 可能我用的DeepSeek V4 Pro/Flash比较傻；
  - 对于一个无关紧要的辅助工程，全部交由他，我越来越不可控；
- 改动方式和设计，还是自己来；
- 码代码，交给AI；
- review和diff，要仔细查看；
- 自定义skill；
  - 限制每次修改前讨论方案；
  - 指出问题后，继续修改给出新的方案；
  - 回答问题不要反问，讨论方案的时候才能反问；
- 需要diff,需要是否应用；

- 理清方案，考虑好每个细节，需要清晰的思路来把控AI的产出；
  - 剩下的就是无聊的实现；

- 在借助开发AI开发的过程中，需要记录&总结的内容需要同步整理；
  - 否则后面就会失去一些掌控力/兴趣；

- 突然的一瞬间，感觉思想有些转变；
  - 在这个时代，并不是要跟AI去做对抗，努力证明自己比AI哪点做的好；
  - 而是需要借助他，让自己变得更好；

> 良好的开发习惯

本地的修改，要定期推送到远程仓库，防止自己误操作，以及AI工具的抽风。

以下总结来自 Gemini:

- AI 工具再强大，也无法代替开发者自身应当具备的工程素养和底层安全防线；
- 保持良好的 Git 提交习惯，不仅是团队协作的要求，也是在各种意外（无论是工具 Bug 还是硬件故障）面前保护自己劳动成果的最有效手段；
